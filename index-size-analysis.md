Stored as a serialized `RoaringBitmap` (u32). Dense bitmap: approximately 4 bytes per tombstone DocId. Sparse: less.


### 1.7 WAL Event Record

From `src/storage/wal.rs:178-182` (append) and `wal.rs:536-562` (encode_event_v2):

```
Frame (8 bytes):
  payload_len: u32 LE  (4 bytes)
  crc32c:      u32 LE  (4 bytes)

Payload (v2/v3, Create event):
  kind:     u8    (1 byte: 1=Create, 2=Delete, 3=Modify, 4=Rename)
  secs:     u64 LE (8 bytes)

```

Create event with 80-byte path: `8(frame) + 1 + 8 + 4 + (1+4+80) + (4+80)` = **~190 bytes on disk**

Rename with two 80-byte paths: `8 + 1 + 8 + 4 + (1+4+80) + (4+80) + (1+4+80) + (4+80)` = **~359 bytes**

## 2. Per-File Index Cost Breakdown

### 2.1 Complete Segment-Level Breakdown

For one file with an 80-byte path, in a diverse corpus of many files:

| Segment | Per-File Cost | Percentage | Notes |
|---------|---------------|------------|-------|
| **PathArena** | 80 bytes | 33.3% | Raw path bytes stored contiguously; root-relative (no root prefix duplication) |
| **Metas** | 40 bytes | 16.7% | Fixed 40B record: dev, ino, root_id, path_off, path_len, size, mtime |
| **TrigramTable** | 6–24 bytes | ~6% | ~0.5–2 *new* unique trigrams per file in large corpus (heavily shared across files) |
| **PostingsBlob** | 80–160 bytes | ~37% | RoaringBitmap posting lists; ~16 trigrams in path × ~5–10 bytes per posting |
| **FileKeyMap** | 24 bytes | 10.0% | Fixed 24B record for live files only |
| **Tombstones** | 0–4 bytes | ~0% | Only when file is deleted; contributes to RoaringBitmap |
| **Container overhead** | ~0 bytes | ~0% | 288B shared across all files → negligible for >1000 files |
| **Total** | **~236–332 bytes** | **100%** | **Typical: ~250 bytes** |

### 2.2 Path Length Sensitivity

How byte cost scales with path length (average, in a large corpus):

| Component | Short (30B path) | Normal (80B) | Long (200B) | Very Long (500B) |
|-----------|-----------------|-------------|-------------|-----------------|
| PathArena | 30 | 80 | 200 | 500 |
| Metas | 40 | 40 | 40 | 40 |
| Avg trigrams/path | ~6 | ~16 | ~40 | ~100 |
| TrigramTable | ~8 | ~16 | ~35 | ~60 |
| PostingsBlob | ~35 | ~100 | ~250 | ~600 |
| FileKeyMap | 24 | 24 | 24 | 24 |
| **Total** | **~137** | **~260** | **~549** | **~1224** |

**Key insight**: Long paths amplify both PathArena (linearly) and trigram posting costs (proportional to component count, ~2.5 trigrams per directory component).

## 3. Segment Structure and Purpose

### 3.1 Seven Segments in Each v6/v7 Container

From `src/storage/snapshot.rs:208-218` (V6SegKind):

| Kind | Name | Purpose | Encoding |
|------|------|---------|----------|
| 1 | **Roots** | Watch root directories | u16 count + per-root (u16 len + bytes) |
| 2 | **PathArena** | All root-relative path strings | Raw concatenated bytes, no per-entry overhead |
| 3 | **Metas** | Per-file metadata records | Fixed 40B records, DocId = array index |
| 4 | **TrigramTable** | Trigram → PostingsBlob index | Sorted 12B entries, binary-searchable |
| 5 | **PostingsBlob** | Concatenated RoaringBitmap posting lists | Variable-length serialized bitmaps |
| 6 | **Tombstones** | Deleted DocIds in this segment | Single serialized RoaringBitmap |
| 7 | **FileKeyMap** | (dev,ino,generation) → DocId | Sorted 24B entries, binary-searchable (with 8B header) |

### 3.2 Segment Details

#### Roots Segment (kind=1)

Encoded as: `u16 count` followed by `count` entries of `(u16 len, bytes)`.

From `src/storage/snapshot.rs:233-254` (`encode_roots_segment`):
- Always starts with `"/"` as first entry
- Other roots are sorted and deduplicated
- Maximum size: `MAX_V6_ROOTS_SEGMENT_BYTES = 1 MB`

Typical size: ~10–50 bytes (1 root path like "/home/user/projects")

#### PathArena Segment (kind=2)

Simple concatenation of all root-relative path bytes. No framing — offsets and lengths come from MetaRecords.

Size = sum of all path lengths.

From l2_partition.rs:1181: `let path_arena_bytes = Arc::clone(&self.arena.read().data);`

#### Metas Segment (kind=3)

Fixed-width 40-byte records. DocId is *implicit* as the array index. If there are N files (including deleted), the segment is exactly N × 40 bytes.

From l2_partition.rs:1193-1209, records are written in DocId order (0, 1, 2, ...):
```rust
let metas = self.metas.read();
let mut metas_bytes = Vec::with_capacity(metas.len() * 40);
for m in metas.iter() {
    metas_bytes.extend_from_slice(&m.file_key.dev.to_le_bytes());
    metas_bytes.extend_from_slice(&m.file_key.ino.to_le_bytes());
    metas_bytes.extend_from_slice(&m.root_id.to_le_bytes());
    metas_bytes.extend_from_slice(&m.path_off.to_le_bytes());
    metas_bytes.extend_from_slice(&m.path_len.to_le_bytes());
    metas_bytes.extend_from_slice(&m.size.to_le_bytes());
    let ns: i64 = m.mtime.and_then(|t| t.duration_since(UNIX_EPOCH).ok())
        .and_then(|d| i64::try_from(d.as_nanos()).ok())
        .unwrap_or(-1);
    metas_bytes.extend_from_slice(&ns.to_le_bytes());
}
```

Tombstoned files still occupy a record slot (their DocId remains allocated) — their entries persist but are masked by the Tombstones bitmap.

#### TrigramTable Segment (kind=4)

Sorted array of 12-byte entries, designed for binary search. Includes a [0,0,0] sentinel entry.

From l2_partition.rs:1243-1258:
```rust
const TRIGRAM_SENTINEL: [u8; 3] = [0, 0, 0];
if !tri_idx.contains_key(&TRIGRAM_SENTINEL) {
    let off: u32 = postings_blob_bytes.len().try_into().unwrap_or(u32::MAX);
    roaring::RoaringBitmap::new()
        .serialize_into(&mut postings_blob_bytes).expect("write to vec");
    let len: u32 = ...
    entries.push((TRIGRAM_SENTINEL, off, len));
}
entries.sort_by_key(|(tri, _, _)| *tri);
```

The sentinel enables `MmapIndex::has_trigram_sentinel()` to distinguish new segments (full-component trigrams) from old segments (basename-only trigrams), avoiding false negatives for directory-name queries.

#### PostingsBlob Segment (kind=5)

Concatenated serialized `RoaringBitmap` (u32) objects. Each trigram's posting list stores the DocIds of files containing that trigram.

Posting list sizes (RoaringBitmap serialization):
- **Empty**: ~5 bytes (empty container marker)
- **1–4096 docids**: ~8 + cardinality × 2 bytes (array container)
- **4097+ docids**: ~1024 bytes per full bitmap container (1024 bytes fixed per 2^16 range)
- **Very large**: multiple containers per trigram, grows with docid range

From l2_partition.rs:1226-1240:
```rust
for (tri, posting) in tri_idx.iter() {
    let off: u32 = postings_blob_bytes.len().try_into().unwrap_or(u32::MAX);
    let posting_bitmap: roaring::RoaringBitmap = posting.iter().map(|v| v as u32).collect();
    posting_bitmap.serialize_into(&mut postings_blob_bytes).expect("write to vec");
    let len: u32 = postings_blob_bytes.len().saturating_sub(off as usize)
        .try_into().unwrap_or(u32::MAX);
    entries.push((*tri, off, len));
}
```

#### Tombstones Segment (kind=6)

A single serialized `RoaringBitmap` of tombstoned DocIds. Read by `MmapIndex::tombstones()` (mmap_index.rs:102-114):
```rust
fn tombstones(&self) -> RoaringTreemap {
    let bytes = self.snap.tombstones_bytes();
    let mut cur = Cursor::new(bytes);
    let b = roaring::RoaringBitmap::deserialize_from(&mut cur)
        .map(|bm| bm.iter().map(|v| v as u64).collect::<RoaringTreemap>())
        .unwrap_or_else(|_| RoaringTreemap::new());
    b
}
```

#### FileKeyMap Segment (kind=7)

Sorted array of (FileKey → DocId) entries. Format with 8-byte header (`FKM\0` + version + flags):

From l2_partition.rs:1287-1321:
```rust
const FKM_MAGIC: [u8; 4] = *b"FKM\0";
const FKM_VERSION: u16 = 1;
const FKM_FLAG_LEGACY: u16 = 0;  // 24B fixed records
const FKM_FLAG_RKYV: u16 = 1;     // rkyv-serialized Vec<FileKeyEntry>

filekey_map_bytes.extend_from_slice(&FKM_MAGIC);
filekey_map_bytes.extend_from_slice(&FKM_VERSION.to_le_bytes());
filekey_map_bytes.extend_from_slice(&FKM_FLAG_LEGACY.to_le_bytes());
// ... then 24B records
```

Read by `MmapIndex::lookup_docid_by_filekey()` (mmap_index.rs:296-384), supporting both legacy (20B) and v6+ (24B) formats, plus the rkyv variant.

### 4.2 Container Overhead Calculation

| Component | Size |
|-----------|------|
| Header (magic + version + state + manifest_len + checksum) | 20 bytes |
| Manifest (magic + version + seg_count + reserved + 7 × 32) | 16 + 224 = 240 bytes |
| Alignment padding (~4 bytes avg × 7 segments) | ~28 bytes |
| **Total per seg-*.db** | **~288 bytes** |

### 4.3 Write Process (Atomic)

From `snapshot.rs:872-1018` (`write_atomic_v6`):

1. Write INCOMPLETE header to `index.db.tmp` (manifest_len/checksum set to 0)
2. Write manifest (segment descriptors with pre-calculated offsets/checksums)
3. Write each segment at its declared offset with 8-byte alignment padding
4. Seek back to offset 0, overwrite with COMMITTED header (real manifest_len + checksum)
5. `fsync(tmpfile)` — ensure data on disk
6. `rename(tmp, target)` — atomic replacement (POSIX guarantee)
7. `fsync(dir)` — ensure directory entry persisted

Checksums: v6 uses `simple_checksum` (rolling add + rotate), v7 uses CRC32C (Castagnoli). Write always produces v7.

### 4.4 LSM Manifest Format

From `snapshot.rs:1023-1030`:

```
LSM_MANIFEST.bin:
  magic:      u32 = 0x314D_534C ("LSM1")
  version:    u32 = 4
  body_len:   u32
  checksum:   u32 (CRC32C for v4+)
  ── body ──
  next_id:         u64    (8 bytes)
  base_id:         u64    (8 bytes)
  delta_count:     u32    (4 bytes)
  delta_ids:       [u64]  (delta_count × 8 bytes)
  wal_seal_id:     u64    (8 bytes)
  last_build_ns:   u64    (8 bytes)
```

Total manifest size: `16 (header) + 36 (fixed body) + delta_count × 8 bytes`.

- 0 deltas: ~52 bytes
- 4 deltas: ~84 bytes
- 7 deltas: ~108 bytes

## 5. Trigram Analysis

### 5.1 Trigram Generation Algorithm

From `src/index/l2_partition.rs:74-88` (`for_each_component_trigram`):

```
For each Component::Normal in the path:
  1. Convert to lowercase
  2. If length >= 3:
     For each 3-byte sliding window:
       Emit the trigram [w[0], w[1], w[2]]
```

This means trigrams are generated from **all path components** (not just basename), covering directory names, file names, and extensions.

### 5.2 Example Path Trigram Enumeration

Path: `/home/user/projects/foo_bar.txt`

| Component | Lowercase | Length | Trigrams | Count |
|-----------|-----------|--------|----------|-------|
| "home" | "home" | 4 | "hom", "ome" | 2 |
| "user" | "user" | 4 | "use", "ser" | 2 |
| "projects" | "projects" | 8 | "pro","roj","oje","jec","ect","cts" | 6 |
| "foo_bar.txt" | "foo_bar.txt" | 11 | "foo","oo_","o_b","_ba","bar","ar.","r.t",".tx","txt" | 9 |
| **Total** | | | | **19 trigrams** |

### 5.3 Trigram Sharing in Large Corpora

For N files with random-ish names of average length L per component:

- Total raw trigrams emitted: approximately N × (avg_total_component_chars - 2 × component_count)
- For 80-byte path (~4 components of ~15 chars each): ~(15-2)×4 = 52 raw trigrams per file
- **Unique trigrams U**: approaches the trigram space size (256³ ≈ 16.7M, but practical alphabet is ~40 chars for lowercase ASCII → ~64K effective). In practice, U saturates quickly:
  - 10K files: U ≈ 25K
  - 100K files: U ≈ 55K
  - 1M files: U ≈ 65K
- **Average posting cardinality**: (N × avg_trigrams_per_file) / U
  - 100K files, 52 trigrams: raw = 5.2M, avg_posting = ~95 docids
  - 1M files, 52 trigrams: raw = 52M, avg_posting = ~800 docids
  - 10M files, 52 trigrams: raw = 520M, avg_posting = ~8000 docids

### 5.4 Posting List Size by Cardinality (RoaringBitmap)

| DocId Count | RoaringBitmap Container | Serialized Size |
|-------------|------------------------|-----------------|
| 0 | Empty | ~5 bytes |
| 1 | Array[1] | ~6 bytes |
| 10 | Array[10] | ~24 bytes |
| 100 | Array[100] | ~204 bytes |
| 1000 | Array[1000] | ~2004 bytes |
| 4096 | Array[4096] (max array) | ~8196 bytes |
| 4097 | 1 bitmap container | ~1030 bytes |
| 65536 | 1 full container | ~1030 bytes |
| 100,000 | 2 containers | ~2050 bytes |
| 1,000,000 | 16 containers | ~16 KB |

Transition at 4096 from array to bitmap is efficient — a 4097-element set is smaller on disk than a 4096-element array.

### 5.5 PostingsBlob Size Estimation

For N files with diverse names:

```
U = min(65536, 65536 × (1 - e^(-N_raw_trigrams/65000)))
avg_posting_cardinality = (N × trigrams_per_file) / U
avg_posting_bytes = 
  if avg_posting_cardinality <= 4096: 8 + 2 × cardinality
  else: (cardinality / 65536) × 1024 (approximate)
Total PostingsBlob = U × avg_posting_bytes
```

Practical estimates:
- 10K files: U ≈ 25K, avg_card ≈ 21, avg_bytes ≈ 50, PostingsBlob ≈ **1.25 MB**
- 100K files: U ≈ 55K, avg_card ≈ 95, avg_bytes ≈ 198, PostingsBlob ≈ **10.9 MB**
- 1M files: U ≈ 65K, avg_card ≈ 800, avg_bytes ≈ 1608, PostingsBlob ≈ **105 MB**
- 10M files: U ≈ 65.5K, avg_card ≈ 8000, avg_bytes ≈ 125, PostingsBlob ≈ **8.2 MB** (bitmap container efficiency kicks in)

Note: The 10M case has low estimates because many DocIds fit within single bitmap containers. Actual size depends on DocId distribution — clustered ranges compress better.

### 5.6 Short Component Index

For 1–2 character path components (e.g., "ab", "x"), the code maintains a separate short-component cache (`MmapIndex::short_component_cache`, mmap_index.rs:492-526). This is built lazily **in memory** (not persisted to disk), mapping from 1-2 character strings to RoaringTreemap posting lists.

The sentinel trigram `[0,0,0]` signals that short-component queries can use the `short_hint_candidates` path (mmap_index.rs:527-556), which:
1. Enumerates trigrams matching the short hint
2. OR-combines their postings
3. Also checks the short component cache for exact 1-2 char matches

## 6. Scale Estimates

### 6.1 Fresh Index (Single Base Segment, No Tombstones)

Assumptions: 80-byte average path, diverse filenames, no deletions.

| File Count | PathArena | Metas | TrigramTable | PostingsBlob | FileKeyMap | Overhead | **Total (approx)** |
|------------|-----------|-------|-------------|-------------|------------|----------|-------------------|
| 1,000 | 80 KB | 40 KB | 60 KB | 120 KB | 24 KB | 0.3 KB | **~324 KB** |
| 10,000 | 0.8 MB | 0.4 MB | 0.3 MB | 1.2 MB | 0.24 MB | 0.3 KB | **~3.0 MB** |
| 100,000 | 8.0 MB | 4.0 MB | 0.7 MB | 10.9 MB | 2.4 MB | 0.3 KB | **~26.0 MB** |

## 8. Compaction Mechanics

### 8.1 Constants

From `src/index/tiered/mod.rs:37-41`:

```rust
const COMPACTION_DELTA_THRESHOLD: usize = 8;
const COMPACTION_MAX_DELTAS_PER_RUN: usize = 4;
const COMPACTION_COOLDOWN: Duration = Duration::from_secs(300);
```

### 8.2 Trigger Conditions

Compaction triggers when `delta_count >= 8` (i.e., `disk_layers.len() - 1 >= 8`).

From `compaction.rs:26-28`:
```rust
let delta_count = layers.len().saturating_sub(1);
if delta_count < COMPACTION_DELTA_THRESHOLD {
    return;
}
```

Only the base + oldest 4 deltas are merged; newer deltas remain as suffix:
```rust
let max_layers = 1 + COMPACTION_MAX_DELTAS_PER_RUN;
if layers.len() > max_layers {
    layers.truncate(max_layers);
}
```

### 8.3 Cooldown

A 5-minute cooldown prevents continuous compaction during high-churn periods:
```rust
let mut g = self.compaction_last_started_at.lock();
if let Some(last) = *g {
    if last.elapsed() < COMPACTION_COOLDOWN {
        return;
    }
}
*g = Some(Instant::now());
```

### 8.4 Standard Compaction (compact_layers)

From `compaction.rs:89-208`:

1. **Verify layer prefix**: Ensure the current disk_layers still begin with the snapshot we captured (handles concurrent append race)
2. **Reject legacy base**: If `layers[0].id == 0`, skip compaction (legacy bases need bootstrap into LSM)
3. **Merge top-down**: Create fresh `PersistentIndex`, feed layers oldest-first:
   ```rust
   for layer in &layers_snapshot {
       layer.deleted_paths.for_each_bytes(|p| {
           let pb = pathbuf_from_bytes(p);
           merged.mark_deleted_by_path(&pb);
       });
       layer.idx.for_each_live_meta(|m| merged.upsert_rename(m));
   }
   ```
4. **Export compacted**: `merged.export_segments_v6_compacted()` → single new base
5. **Atomic replace**: `store.replace_base_v6(&segs, Some((base_id, delta_ids)), ...)` — CAS on manifest prefix
6. **GC**: `store.gc_stale_segments()` removes old seg-*.db/.del files
7. **Update disk_layers**: Replace merged prefix with new base, keep suffix deltas

### 8.5 Fast Compaction (compact_layers_fast)

Enabled by `FAST_COMPACTION=1`. From `compaction.rs:210-371`:

Instead of feeding through `PersistentIndex::upsert_rename` (which rebuilds trigram index from scratch), the fast path:

1. **Collect top-down, deduplicate by FileKey** (newest wins):
   ```rust
   for layer in layers_snapshot.iter().rev() {
       layer.idx.for_each_live_meta(|m| {
           let new_docid = final_metas.len() as DocId;
           if seen_fk.insert(m.file_key) {
               mapping.push(Some(new_docid));
               final_metas.push(m);
           } else {
               mapping.push(None);
           }
       });
       layer_mappings.push(mapping);
   }
   ```
2. **Remap trigram postings**: Rebuild trigram index by translating old DocIds → new DocIds
3. **Apply deleted paths** after trigram remapping
4. **Bulk fill**: `merged.fill_from_compaction(roots, final_metas, merged_trigrams)`
5. Same atomic replace + GC

### 8.6 Dead Space Reclaimed by Compaction

- **Duplicate FileKey entries**: Multiple layers may have entries for the same file (created in delta, also in base). Only the newest survives.
- **Obsolete tombstones**: When a file is deleted in delta N, its tombstone persists. Compaction removes both the base entry and the tombstone.
- **Duplicate trigram postings**: Each layer has its own trigram posting lists. Merged into one set.
- **Old segment files**: Physical `seg-*.db` and `seg-*.del` files removed from disk.

### 8.7 Compaction Efficiency Ratio

For base + 4 deltas where each delta modifies ~15% of base files:
- **Before**: size ≈ base + 4 × 0.15 × base = **1.6 × base**
- **After**: size ≈ base (fully deduplicated) = **1.0 × base**
- **Reclaimed**: ~37.5% of total size

For base + 4 deltas where each delta is a full snapshot (degraded watcher):
- **Before**: size ≈ 5 × base = **5.0 × base**
- **After**: size ≈ base = **1.0 × base**
- **Reclaimed**: ~80% of total size

## 9. Growth Factors

### 9.1 LSM Delta Segment Growth

Each `snapshot_now()` creates a delta segment containing:
1. All files created/modified since the last snapshot (new delta's seg-*.db)
2. All files deleted since the last snapshot (new delta's seg-*.del sidecar)
3. Internal tombstones for files deleted within this specific segment

Delta segments are **independent snapshots** — no cross-segment data sharing. A file appearing in multiple deltas has duplicate entries until compaction.

### 9.2 Tombstone Accumulation

Two forms of tombstones:

**Internal (Tombstones segment)**: RoaringBitmap within each seg-*.db. Masks DocIds that were deleted *within that segment's own set*. Cost: ~4 bytes per tombstone.

**Cross-segment (DEL sidecar)**: `seg-*.del` files containing path strings. Masks entries in *all previous layers* (base and older deltas). Cost: 12 + D × (2 + P) bytes.

For D deleted paths in a delta:
- Internal tombstone cost: ~4D bytes (in seg-*.db)
- DEL sidecar cost: 12 + D × (2 + P) bytes
- **Total tombstone cost per deletion**: ~6 + P bytes

With D=10,000 and P=80: ~86 bytes per deletion = ~860 KB per delta.

### 9.3 WAL Growth Between Snapshots

WAL accumulates events between snapshots. Growth rate:

```
WAL_growth = events_per_interval × avg_event_size
```

Typical event rates:
- Idle system: ~0–10 events/min → negligible
- Active development (compile cycle): ~100 events/min → ~19 KB/min → ~1.1 MB/hour
- Mass file operation (untar, git checkout): ~10K events → ~1.9 MB in one burst
- Full filesystem scan: ~100K+ events → ~19+ MB

The WAL is sealed at each snapshot boundary and cleaned up after successful flush.

### 9.4 Cumulative Index Growth Over Time

```
Time 0:  Base only         = N × 250 bytes
Time 1:  Base + 1 delta    = N × 250 + 0.15N × 250    = N × 287
Time 2:  Base + 2 deltas   = N × 250 + 0.30N × 250    = N × 325
...
Time 7:  Base + 7 deltas   = N × 250 + 1.05N × 250    = N × 512
Time 8:  Compaction fires  → Base only                 = N × 250
```

Steady state (15% churn per interval, compaction at 8 deltas):
- Average size: approximately N × 380 bytes
- Peak size (just before compaction): N × 512 bytes
- Post-compaction: N × 250 bytes
- **Average amplification**: ~1.5× over base size

## 10. WAL Growth Analysis

### 10.1 Per-Event Disk Cost

From `src/storage/wal.rs:170-185` (append) and encoding:

| Event Type | Payload Fields | Payload Size (80B path) | Frame (8B) | **Total (80B path)** |
|------------|---------------|------------------------|------------|---------------------|
| Create | kind(1)+secs(8)+nanos(4)+file_id(1+4+80)+hint(4+80) | 182 | 8 | **190 bytes** |
| Delete | same as Create | 182 | 8 | **190 bytes** |
| Modify | same as Create | 182 | 8 | **190 bytes** |
| Rename | Create + from_id(1+4+80) + from_hint(4+80) | 351 | 8 | **359 bytes** |

### 10.2 WAL File Format

```
events.wal:
  [MAGIC:    4 bytes = 0x314C_4157 "WAL1"]
  [VERSION:  4 bytes = 3]
  ── Records (repeat) ──
  [payload_len: u32 LE]
  [crc32c:      u32 LE]
  [payload:     payload_len bytes]
  ── End ──
```

The file grows append-only. At seal time, it's renamed to `events.wal.seal-<timestamp_nanos>` and a new empty WAL is created.

### 10.3 WAL File Lifecycle

1. Events appended to `events.wal` throughout an epoch
2. `snapshot_now()` calls `wal.seal()`:
   - Flushes current WAL
   - Renames to `events.wal.seal-<id>`
   - Creates fresh `events.wal`
   - Returns `seal_id` (Unix epoch nanos)
3. After successful delta append, `cleanup_sealed_up_to(wal_seal_id)` removes sealed WALs with id ≤ wal_seal_id
4. On cold restart, `replay_since_seal(checkpoint_seal_id)` replays only WALs with id > checkpoint

### 10.4 WAL Safety

From `wal.rs:14`: `const MAX_WAL_RECORD_BYTES: usize = 8 * 1024 * 1024; // 8 MiB`

Individual records are capped at 8 MB to prevent memory DoS from corrupted files.

Each record has a CRC32C checksum. Corrupted records are skipped during replay (counted in `truncated_tail_records`).

### 10.5 WAL Accumulation Scenarios

| Scenario | Events/Interval | Interval | WAL Growth |
|----------|----------------|----------|------------|
| Idle, 60s snapshots | 5 | 60s | ~1 KB/interval |
| Coding, 60s snapshots | 200 | 60s | ~38 KB/interval |

## 12. Pathological Growth Scenarios

### 12.1 Many Small Files with Long Paths

**Scenario**: 1M files, each with 200-byte paths in deep directory trees (e.g., Java/Maven projects).

| Component | Normal (80B path) | Long (200B) | Blowup Factor |
|-----------|------------------|-------------|---------------|
| PathArena | 80 MB | 200 MB | 2.5× |
| Trigram count per file | ~16 | ~40 | 2.5× |
| TrigramTable | ~0.8 MB | ~1.5 MB | 1.9× |
| PostingsBlob | ~105 MB | ~250 MB | 2.4× |
| Metas | 40 MB | 40 MB | 1.0× |
| FileKeyMap | 24 MB | 24 MB | 1.0× |
| **Total** | **~250 MB** | **~516 MB** | **2.1×** |

**Root cause**: Each additional directory component adds ~3 trigrams. Deep trees (10+ levels) can produce 40+ trigrams per file, significantly expanding the PostingsBlob.

**Mitigation**: Regular compaction keeps the base segment compact. The trigram index is shared efficiently — only *new* trigrams add to TrigramTable.

### 12.2 Rapid Create/Delete Cycles (Churn)

**Scenario**: Create 10K temp files, delete all, repeat 10× every minute.

```
Cycle 0: Base: 10K files → ~2.5 MB
Cycle 1: Delta: creates (~2.5 MB) + tombstones + DEL (~0.86 MB) → +3.36 MB
Cycle 2-7: Each adds ~3.36 MB
Cycle 7 total: Base + 7 deltas = 2.5 + 7×3.36 = ~26 MB
Cycle 8: Compaction triggers → compacts to Base = ~0 MB (all files deleted)
```

**Worst case** (compaction never fires): unlimited growth, ~3.36 MB per cycle.

**With normal compaction**: After cycle 8, compacts to a tiny base (tombstone-only, ~<1 KB).

### 12.3 Many Rename Operations

**Scenario**: 50K files renamed per minute in a watched directory.

Each rename = 1 overlay entry. At snapshot:
1. Old path → added to deleted_paths overlay (path bytes copied to Arena)
2. New path → upserted to L2 (new MetaRecord + trigrams + arena entry)
3. Old in-segment entry → not tombstoned internally (rename preserves FileKey)

Effect on next delta segment:
- If path components change significantly: ~250 bytes per renamed file (≈ same as new file)
- If basename only changes: slightly less (path shared until last component)
- Old entry in base remains, masked by DEL sidecar → space waste until compaction

**Compaction recovery**: Removes the masked base entry, keeping only the newest path. Net cost after compaction: ~250 bytes per file (the live version).

### 12.4 Degraded Watcher → Many Delta Segments

**Scenario**: Inotify queue overflows, watcher falls back to periodic full filesystem scans. Each scan creates a complete delta segment.

```
Base (initial scan):    ~250 MB (1M files)
Delta 1 (scan 2):       ~250 MB (full delta)
Delta 2 (scan 3):       ~250 MB
...
Delta 7 (scan 8):       ~250 MB
Pre-compaction total:   ~2.25 GB (9× blowup)
After compaction:       ~250 MB (merged to one base)
```

The 5-minute compaction cooldown prevents *continuous* compaction during degraded periods, but the 8-delta threshold keeps the blowup bounded at ~9×.

**Note**: `MIN_SNAPSHOT_INTERVAL = 10 seconds` (snapshot.rs:15) further limits snapshot frequency, giving at least 80 seconds between compaction cycles during degraded mode.

### 12.5 Maximum Deleted Path Accumulation

Safety limit: 500,000 paths (snapshot.rs:1029). At 80 bytes per path:

- DEL sidecar: 12 + 500K × (2 + 80) = **~41 MB** in worst case
- Internal tombstones: 500K × 4 = **~2 MB**
- Total deletion overhead: **~43 MB**

Beyond this, new deletions would be rejected (safety guard). Compaction clears these by merging tombstones into the base.

## 13. Index File Sizes: Rules of Thumb

| Metric | Formula |
|--------|---------|
| **Base index size** | N × 200–300 bytes (≈ N × 250 bytes typically) |
| **With 4 deltas (typical steady state)** | N × 250 × 1.4 = N × 350 bytes |
| **With 7 deltas (pre-compaction peak)** | N × 250 × 1.8 = N × 450 bytes |
| **Post-compaction** | N × 250 bytes |
| **Per-file in base only** | PathArena: P bytes, Metas: 40, Trigram: ~15, Postings: ~0.12N/P (amortized), FileKeyMap: 24 |
| **Path overhead per extra 100 chars** | +100 bytes PathArena + ~60 bytes trigrams = +160 bytes |
| **Per deleted file (in DEL sidecar)** | 12/D + 2 + P ≈ P+2 bytes per file (~82 for P=80) |
| **Per seg-*.db file overhead** | 288 bytes (header + manifest + alignment) |
| **Per seg-*.del file overhead** | 12 bytes header + entries |
| **WAL per snapshot epoch** | events × 190 bytes (Create/Delete/Modify) or × 359 bytes (Rename) |
| **Index:path ratio** | ~3.2:1 (80-byte path → ~250-byte index) |
| **Index:file_size ratio (avg 10KB files)** | ~0.025:1 |
| **Index:file_size ratio (avg 1MB files)** | ~0.00025:1 |
